Ato 1 · O problema

Não é só trocar ferramenta

❌ Antes — Centralizado
  • PM escreve todas as issues
  • PM define critérios de aceite
  • PM cria histórias e sub-tasks
  • Time executa o que o PM definiu
✓ Depois — Distribuído
  • PM define a direção macro
  • Designer escreve suas próprias issues
  • Engenheiro escreve as issues técnicas
  • Cada um é dono do que sabe fazer

É trocar mentalidade. Autonomia, senso de dono, decisão perto de quem executa.

Ato 2 · A virada

Por que o Linear?

"Ferramenta construída para os criadores — não para gerar relatórios perfeitos."

— Linear Method

Decisão

Estratégica, não comparação técnica de features

Foco

Autonomia real para quem constrói o produto

Cultura

Ownership distribuído desde a base

Ato 2 · A virada

Os princípios do Linear Method

01
Issues ≠ User Stories

Chega de "Como usuário, eu quero...". Issues descrevem tarefas concretas, direto ao ponto, sem burocracia de template.

02
Escreva para quem executa

Quem entende o trabalho escreve a issue. Sem intermediário, sem tradução — direto da fonte para a execução.

03
Momentum, não sprint heroico

Escopo pequeno, cadência constante. Entrega contínua supera o acúmulo seguido de sprint de apagar incêndio.

04
Direção significativa

Todo trabalho conectado a um propósito maior. Iniciativas e Milestones lembram o time por que o trabalho existe.

Ato 3 · O mapa

Os dois caminhos: Discovery e Delivery

Caminho A — Projeto de Discovery
  • 1.Escopo indefinido — PM cria briefing (não spec)
  • 2.Designer explora livremente
  • 3.M1 Direção validada — PM fecha spec
  • 4.M2 Spec aprovada → Build começa

Use quando o problema é claro, mas o escopo ainda será definido com o Designer.

Caminho B — Discovery como Milestone
  • 1.Problema claro — PM cria spec desde o início
  • 2.M1 Discovery concluído — direção aprovada
  • 3.M2 Spec final → Engenharia entra

Use quando o discovery é uma fase curta e bem delimitada dentro de um projeto maior.

⚠ Anti-pattern: Esperar o escopo estar completo para criar o projeto = discovery sem estrutura, sem milestone, sem contexto compartilhado.

Ato 3 · O mapa

O que o PM faz no Linear

Regra de ouro — títulos

[Verbo imperativo] + [Ação] + [Valor para o negócio]

❌ Ruim
Acompanhamento do professor
✓ Bom
Garantir visibilidade do professor sobre envios das equipes

Briefing vs. Spec

Briefing Discovery aberto — problema + critérios de saída
Spec Build — escopo fechado, critérios de aceite binários

Use /linear-spec para estruturar qualquer um dos dois com o assistente treinado no seu processo.

Ato 4 · A prática

Como fazer hoje

Para o PM
/linear-spec

Assistente para estruturar Iniciativas, Projetos e Milestones. Já conhece o processo do nosso time e a regra de ouro de títulos.

  • /linear-spec create — nova Iniciativa ou Projeto
  • /linear-spec brief — briefing para discovery
  • /linear-spec validate — validar item existente
Para o time
/linear-issues

Assistente para estruturar issues e sub-issues. Garante que issue pai foca em valor e sub-issues têm prefixos técnicos corretos.

  • /linear-issues create — estruturar nova issue
  • /linear-issues validate CLMS-1234 — validar issue
  • /linear-issues help — exemplos e boas práticas
💡

Ambos os skills rodam direto no Claude Code — no mesmo ambiente onde você já trabalha.
