Diagrama de Casos de Uso · Elementos

Três elementos. Um propósito: quem faz o quê.

🧑

(externo)

Outro Caso de Uso
Ator: qualquer entidade externa — usuário humano, outro sistema, hardware. "Quem usa."
Caso de Uso (oval): uma funcionalidade que o sistema oferece ao ator. Sempre um verbo: "Abrir Conta", não "Conta".
Sistema (retângulo): o limite — o que é responsabilidade do sistema vs. o que é externo.

ATO 3 · Design Thinking

A maioria dos times modela a solução errada para o problema errado.

Não porque são ruins — porque nunca foram ensinados a empatizar antes de modelar.

Design Thinking é o antídoto.

Design Thinking · Processo

5 fases. Não é linear — é iterativo.

01EmpatizarObserve o usuário. Sinta a dor.
02DefinirQual é o problema real?
03IdearGere soluções. Sem censura.
04PrototiparConstrua algo tangível rápido.
05TestarValide com usuários reais.

Você volta ao início quando descobre que estava resolvendo o problema errado. Isso é saúde no projeto, não falha.

Design Thinking · Aplicado ao Nubank

Como o Nubank usou cada fase.

Empatizar: David foi ao banco e viveu a burocracia na pele. Mapeou a frustração como usuário.
Definir: O problema não é "banco ruim" — é "abrir conta é humilhante e burocrático". Foco no processo.
Idear: 100% digital, sem agência, sem documentos físicos, sem anuidade no cartão.
Prototipar + Testar: → próximo bloco.

O produto surgiu da fase Definir — não da fase Idear. Definir o problema certo foi o diferencial competitivo do Nubank.

Ponte · PRD

Design Thinking mapeou o problema. Quem garante que todo o time entende o mesmo problema?

A falta de um documento claro é a causa número 1 de retrabalho em produto digital.

Esse documento se chama PRD — Product Requirements Document.

PRD · Product Requirements Document

O PRD muda de forma dependendo da fase do projeto.

🔍
Briefing (Discovery)

Escopo ainda indefinido. Problema claro.

Foco: dor do usuário + perguntas em aberto
Saída: designer inicia exploração
📋
Spec Completa (Build)

Direção validada. Escopo fechado.

Foco: critérios de aceite binários e testáveis
Saída: engenharia inicia o build

Princípio: O projeto não precisa ter escopo completo para existir. Ele precisa ter um problema claro.

PRD · Anatomia do Briefing

Um bom briefing responde três perguntas.

🧠 Por quê? — o problema, a dor do usuário, os dados que justificam.
Ex: "Abrir conta levava 40 dias. 60% dos usuários desistiam no processo."
🎯 O quê? — escopo: o que está dentro ✅ e o que está fora 🚫.
Ex: ✅ Abertura 100% digital  ·  🚫 Conta PJ (fora do MVP)
✅ Como saber que está pronto? — critérios de aceite binários e testáveis.
Ex: "4 de 5 usuários abrem conta sem ajuda em menos de 5 minutos."

O PM que escreve um bom briefing economiza semanas de retrabalho — e evita que o time construa a solução certa para o problema errado.
